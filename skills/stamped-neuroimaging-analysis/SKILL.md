# Coordinate a STAMPED neuroimaging analysis

Use the local [STAMPED principles paper](../../../Macdonald%20et%20al.%20-%20STAMPED%20principles%20for%20reproducible%20research%20objects.pdf), or its [canonical source](https://github.com/stamped-principles/stamped-paper), as the normative basis. Do not restate the principles. Apply them through the concrete responsibilities and gates below.

## Assign one authority to each concern

| Concern | Authority |
|---|---|
| Python package metadata | Root `pyproject.toml` |
| Local development and tool environments | `envs/pixi.toml` plus `envs/pixi.lock` |
| Result-producing runtime | An exact, tracked SIF image built from a digest-identified OCI image |
| Code, configuration, and small metadata history | Git/DataLad |
| Large data and container content identity | git-annex through DataLad |
| Commands that create or change scientific results | `datalad run`, `datalad containers-run`, or the DataLad run records generated by a pinned BABS release |
| Subject/session fan-out and Slurm operations | BABS |
| BIDS composition | A study dataset that links raw inputs and independent derivative datasets |

Never claim that a Pixi lock identifies a result-producing runtime. The SIF content used by the run is that identity. A lock is a reviewed build input and a convenient local environment specification.

## Establish the research-object boundary

1. Identify the top-level DataLad dataset and, if applicable, the exact result set to reproduce.
2. Inventory code, input datasets, derivative datasets, container datasets, configuration, licenses, and external retrieval locations.
3. Replace untracked absolute paths with paths relative to the dataset root or with explicit host configuration that is excluded from scientific parameters.
4. Resolve unresolved access, license, version, or cohort questions. Do not silently infer them.

Use this composition unless the project already has a stricter compatible layout:

```text
research-object/
├── code/ or src/                 # importable code and explicit CLIs
├── config/                       # scientific parameters and BABS YAML
├── envs/                         # Pixi files, image recipes, image ledger
│   └── containers/               # DataLad container subdataset or its registration
├── operations/                   # exact BABS lifecycle commands and campaign records
├── studies/
│   └── <study>/
│       ├── sourcedata/<input>/   # DataLad subdataset or tracked retrieval adapter
│       └── derivatives/<run>/    # independent DataLad/BABS result dataset
├── tests/
├── docs/
└── skills/stamped-neuroimaging-analysis/
```

Keep each raw dataset, container collection, and derivative independently versioned. Compose them from the study or superdataset. Never place a derivative inside the raw dataset that produced it.

## Isolate open and controlled data

- Install a public BIDS input as a pinned DataLad subdataset or snapshot, not as an unversioned download.
- For controlled data, track only allowed metadata, stable access URLs or retrieval adapters, and the precise selection procedure in the public object. Put content and sensitive provenance on access-controlled DataLad siblings.
- Treat derivative redistribution as a separate policy decision. Do not assume derived morphometrics are shareable because code is shareable.
- Keep public and controlled studies in separate DataLad datasets and storage siblings. Prevent a recursive public push from reaching controlled annex content.
- Record the dataset description, license or DUC, snapshot/commit, retrieval method, and cohort-generation command before processing.

Stop and request a policy decision if a proposed retrieval, log, derivative, or subject-level record may violate a DUA/DUC.

## Make scientific code explicit

- Move authoritative processing, extraction, statistics, and figure generation into importable modules with non-interactive CLIs.
- Give every command explicit input, output, configuration, and random-seed arguments.
- Keep notebooks only for exploration, explanation, or review. Do not make notebook state the sole implementation or provenance of a result.
- Separate participant processing, morphometric extraction, cohort assembly, modeling, figure generation, and validation. Record each scientific boundary separately.
- Put scientific parameters in tracked configuration. Keep scheduler resources and site paths in separate host profiles.

## Manage Pixi without making it a scientific workflow engine

- Keep every Pixi environment in `envs/pixi.toml` and commit `envs/pixi.lock`.
- Track root symlinks to `envs/pixi.toml`, `envs/pixi.lock`, and `envs/.pixi`. Ignore realized `envs/.pixi/` contents and `.venv/`.
- Define distinct named environments when dependencies conflict. Do not force incompatible extraction, analysis, BABS, and development stacks into one solve.
- Change the manifest and lock only during an explicit dependency-authoring session. Use `--locked` for ordinary realization and testing.
- Never use `pixi pack` or `pixi-pack` for a scientific runtime or distribution artifact.
- Never use Pixi task dependencies.
- Never use a Pixi task to process inputs, create derivatives, select a cohort, fit a model, generate an authoritative figure, submit or merge a BABS campaign, or perform any other action that bears on the scientific analysis.
- Limit Pixi tasks to development conveniences and read-only aids such as linting, unit tests, documentation checks, environment reports, `babs status`, configuration validation, or printing an exact command from a tracked runbook.

When a local scientific step needs a Pixi-only development environment, pass the explicit executable and all arguments directly to `datalad run`; do not record only a Pixi task name. Production and release steps must use the tracked SIF.

## Build and register every scientific runtime

For each processing or analysis runtime:

1. Track its OCI recipe and build context under `envs/containers/<runtime>/`.
2. Pin every base image by OCI digest.
3. Copy the reviewed Pixi manifest and lock into the build and realize them with `--locked`. Use Pixi as a builder; the final image is the runtime.
4. Build and test the OCI image in clean Linux automation. Record the builder image, Pixi version, target platform, recipe commit, lock hash, OCI digest, and smoke-test result.
5. Sign and verify the OCI digest with the project’s Sigstore identity policy. Attach an SBOM and provenance attestation where supported.
6. Convert that exact OCI digest to SIF with a pinned Apptainer/Singularity version. Record the conversion command and SIF SHA-256 separately because repeated OCI-to-SIF conversion need not be byte-identical.
7. Store the SIF in a registry and as annexed content in a DataLad container dataset. Register the actual SIF with `datalad containers-add`; do not register only a mutable tag.
8. Track the SIF retrieval locations, annex key, SHA-256, signature bundle or Apptainer signature, source OCI digest, and supported host architecture in the image ledger.
9. Keep required license files or credentials outside the image and outside Git. Document and validate the runtime bind mount.

Do not run an HPC scientific job unless its exact SIF is already tracked, retrievable, and verified. A recipe, Pixi lock, OCI tag, or node-local conversion cache is not sufficient.

## Execute through DataLad and BABS

Use `datalad containers-run` for direct containerized scientific steps when practical. Declare the exact SIF, code/configuration, data inputs, and outputs. Ensure the recorded command exposes the scientific executable and critical arguments.

Use BABS for participant/session expansion, Slurm submission, status/audit, retry, and merge:

1. Pin BABS and DataLad versions in a dedicated Pixi environment.
2. Give BABS a pinned BIDS input DataLad dataset, a DataLad container dataset containing the exact SIF, and a tracked container configuration YAML.
3. Create one derivative DataLad dataset per dataset × pipeline × input-condition campaign. Use stable, version-bearing campaign identifiers.
4. Run a one-participant setup test and inspect the generated job script, run record, input dataset commit, container key, bind mounts, output layout, and resource request before submitting a campaign.
5. Record the exact lifecycle commands—`babs init`, setup checks, submit, status-driven retries, and merge—in `operations/<campaign>/`. Include tool versions, configuration hashes, timestamps, actor, and before/after dataset commits.
6. Accept the current BABS wrapper/zip layer only if the generated scripts are tracked, the SIF and inputs are exact, and a representative historical replay is tested. Do not describe the wrapper name alone as transparent command provenance.
7. Record unzip/materialization as its own `datalad run` step. Treat the zip as a tracked result, not an inconvenience to bypass outside provenance.

Do not require unreleased BABS branches. Reassess upstream `datalad containers-run` integration when it is released and validated; until then, document the remaining indirection.

## Keep HPC execution disposable and portable

- Target Slurm through BABS and isolate site-specific partitions, accounts, scratch roots, and module/bootstrap commands in tracked host profiles.
- Use node-local scratch as disposable compute space. Persist only declared outputs, logs needed for audit, and provenance records.
- Bind data read-only unless a declared output requires writes. Bind the FreeSurfer license or other credentials at runtime.
- Do not bind a developer `.venv`, Pixi prefix, home directory, or untracked package cache into a scientific job.
- On macOS, use Pixi for local development and use a Linux VM such as Lima only to build or smoke-test the same Linux container path. Do not treat a macOS environment as evidence that the HPC SIF is reproducible.

## Validate before scaling or releasing

Require all of the following:

- A clean recursive clone can obtain every permitted component or gives a precise access-controlled retrieval instruction.
- Every result-changing commit has a DataLad run record or a documented BABS-generated run record.
- Every run resolves to exact code, configuration, data commits, and SIF content.
- `datalad rerun --report` is intelligible, and representative processing and downstream analysis steps replay from historical state.
- A clean SIF execution reproduces the small-fixture outputs and passes domain checks.
- BIDS input and derivative datasets validate to the declared standard or carry reviewed exceptions.
- Container signatures, SIF checksums, SBOMs, and retrieval paths verify.
- Public release tests cannot fetch controlled annex content.
- Code, documentation, data, models, and containers have independent, compatible license/DUC records.
- The result manifest maps each authoritative figure, table, and statistic to its generating commit and files.

Report failed gates with evidence. Do not submit a larger campaign to compensate for an unresolved pilot failure.
